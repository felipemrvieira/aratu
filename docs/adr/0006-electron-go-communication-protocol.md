# ADR 0006 — Electron ↔ Go Communication Protocol

**Status:** Accepted  
**Date:** 2026-06-22

## Context

Electron main must supervise and communicate with the local Go engine without exposing process, network, credential, or database capabilities to the renderer. The protocol needs to remain understandable during early development, testable outside Electron, compatible with packaged desktop builds, and secure against unintended local clients.

The first engine workflows are bounded control and data operations: health, engine information, connector discovery, connection tests, schema introspection, paginated previews, query execution, cancellation, and shutdown. High-volume streaming may become relevant later, but its requirements are not yet proven.

## Decision

Use:

- **HTTP/JSON** over IPv4 loopback as the Electron main ↔ Go engine transport;
- an operating-system-assigned ephemeral port requested with port `0`;
- a single structured readiness event on `stdout` for endpoint discovery;
- a per-run 256-bit bearer token generated by Electron main and delivered to the child through `stdin`;
- **JSON Schema Draft 2020-12** as the source of truth for shared payload contracts;
- named, allowlisted Electron IPC/preload methods between renderer and main;
- bounded pagination for initial result delivery, with streaming deferred until measured requirements justify it.

Electron main is the only engine client in the application architecture. The renderer must never receive the engine port, bearer token, generic HTTP access, or unrestricted IPC access.

## Process Bootstrap

### Electron main responsibilities

1. Resolve and verify the engine binary appropriate to the current platform and packaged/development mode.
2. Generate 32 cryptographically secure random bytes and encode them as an unpadded base64url token.
3. Spawn the engine with a constrained environment and `--port=0`. The engine always binds to `127.0.0.1`; a general network-bind option is not exposed in the desktop distribution.
4. Write one newline-delimited bootstrap object to the child's `stdin`, then close the stream:

```json
{"type":"bootstrap","token":"<base64url-token>"}
```

5. Await one newline-delimited readiness event from `stdout` within a startup deadline.
6. Validate the event shape and supported protocol version.
7. Call the authenticated health endpoint before exposing engine-backed preload capabilities.

### Engine responsibilities

1. Read and validate the bootstrap object before becoming ready.
2. Reject missing, malformed, short, or additional bootstrap input.
3. Bind only to `127.0.0.1` with an ephemeral port.
4. Initialize required services and begin accepting authenticated requests.
5. Write exactly one readiness event to `stdout`:

```json
{
  "event": "ready",
  "host": "127.0.0.1",
  "port": 43127,
  "protocolVersion": "1",
  "engineVersion": "0.1.0"
}
```

6. Write structured operational logs only to `stderr`. No log entry may contain the bootstrap token, authorization header, connection credential, or unrestricted query result.

Unexpected output, premature exit, timeout, unsupported protocol version, or failed health validation makes startup fail closed.

## Local Authentication

Every HTTP endpoint, including health and shutdown, requires:

```http
Authorization: Bearer <per-run-token>
```

The engine compares tokens without data-dependent early exit where supported by the selected implementation. Requests without valid authentication receive a generic unauthorized response and no operational detail.

Additional controls:

- bind exclusively to `127.0.0.1`, never `0.0.0.0` or a LAN interface;
- do not enable browser CORS;
- validate expected HTTP methods, content types, body limits, and host/authority values;
- never place the token in CLI arguments, files, logs, URLs, renderer state, or persistent storage;
- discard the token when the process exits;
- generate a new token and port for every engine process.

Loopback is a network isolation boundary, not an authentication mechanism. The bearer token protects the sidecar from other local applications that discover its port.

## Contract Source of Truth

Shared request, response, event, and error payloads live under `packages/contracts/schemas/` as JSON Schema Draft 2020-12 documents.

Each public schema must define:

- a stable `$id` under the Aratu contract namespace;
- an explicit title and description;
- closed objects using `additionalProperties: false` unless extensibility is intentional;
- required versus optional fields;
- bounds for strings, arrays, numbers, and nested payloads;
- examples kept separately under `packages/contracts/examples/`;
- secret and sensitive-data handling notes where relevant.

TypeScript types and Go representations should be generated from or mechanically checked against these schemas. Runtime validation occurs at trust boundaries. The exact code-generation and validation tools are deferred until the TypeScript and Go workspaces are initialized; manually duplicated authoritative models are not allowed.

JSON Schema defines payload shapes, not HTTP routes. Endpoint behavior, methods, status codes, authentication, idempotency, and lifecycle semantics are documented in the contract package and tested as protocol behavior. OpenAPI is not selected as a second source of truth in this phase; it may later be generated or adopted through a separate decision if the API surface justifies it.

## Initial Protocol Surface

The initial route plan is:

```txt
GET    /v1/health
GET    /v1/engine
GET    /v1/connectors
POST   /v1/connections/test
POST   /v1/connections
DELETE /v1/connections/{connectionId}
POST   /v1/operations/{operationId}/cancel
POST   /v1/shutdown
```

Routes beyond health and engine information remain planned contracts, not implemented features. Database discovery and query routes will be added only with their vertical slices.

Successful responses use resource-specific JSON objects rather than a universal success envelope. Errors use one stable envelope:

```json
{
  "error": {
    "code": "CONNECTION_TIMEOUT",
    "message": "The data source did not respond before the timeout.",
    "retryable": true,
    "requestId": "req_01J...",
    "details": {}
  }
}
```

Error messages are suitable for users but reveal no credentials, authorization data, raw driver secrets, stack traces, or sensitive values. `details` is bounded, code-specific, and schema-validated.

## Request Identity, Timeouts, and Cancellation

- Electron main assigns every request a unique `X-Aratu-Request-Id`; the engine returns it in the response and safe logs.
- Every engine operation has a configured deadline. The client transport also has finite connect, header, body, and overall timeouts.
- Potentially long-running work returns or exposes an `operationId` associated with a cancelable Go `context.Context`.
- `POST /v1/operations/{operationId}/cancel` is idempotent and reports whether the operation was running, already completed, or unknown.
- Client disconnect, explicit cancellation, engine shutdown, and applicable database-driver cancellation propagate to the operation context.
- The application never retries mutation, staged apply, or ambiguous operations automatically. Read-only retries require a later explicit idempotency policy.

## Result Delivery

Initial data APIs use bounded pages with explicit limits and opaque continuation state where the source supports it. The engine enforces maximum row/document/key counts and maximum serialized payload size independently of client input.

HTTP streaming, NDJSON, Server-Sent Events, WebSocket, or gRPC streaming are not part of this decision. If measurements show that bounded pages are insufficient for exports, logs, or large values, a later ADR must define backpressure, cancellation, partial failure, memory limits, and compatibility behavior.

## Version Compatibility

`protocolVersion` represents breaking compatibility between the packaged desktop client and engine protocol. The initial value is `1`.

- Electron declares the protocol versions it supports.
- The engine reports its protocol and engine versions in readiness and engine information.
- Unsupported combinations fail before connection or query operations become available.
- New optional fields require schema and compatibility review; clients must not assume that adding a field is harmless when objects are closed.
- The packaged desktop release must contain the engine version it was tested against. Substituting arbitrary external engine binaries is unsupported.

A detailed backward-compatibility window is deferred until more than one released protocol version exists.

## Lifecycle

### Startup

Electron spawns one engine, sends bootstrap authentication, validates readiness, checks compatibility, and calls health. Startup has a finite deadline and produces a user-visible unavailable state on failure.

### Running

Electron main monitors the child process and owns all engine calls. Renderer requests pass through allowlisted preload and IPC methods with schema validation. Database sessions are engine-owned and referenced through opaque identifiers.

### Graceful shutdown

1. Electron stops accepting new renderer operations.
2. Electron calls authenticated `POST /v1/shutdown` with a finite grace period.
3. The engine marks itself draining, rejects new operations, cancels or awaits existing work according to operation policy, and closes data-source sessions.
4. The engine exits with an intentional status.
5. Electron sends an operating-system termination signal if the grace deadline expires and force-kills only as the final fallback.

### Unexpected exit

- The UI moves to an engine-unavailable state and invalidates engine-owned session identifiers.
- No query, mutation, staged apply, or cancellation request is replayed automatically.
- At most one automatic restart may be attempted when no mutation outcome is ambiguous.
- Repeated failure enters a stopped state and requires explicit user action or application restart.
- Restart behavior must be covered by integration tests before it is enabled.

## Alternatives Considered

### gRPC

Provides protobuf contracts, client generation, bidirectional streaming, deadlines, and cancellation. It was not selected because it adds code-generation and Electron/Node transport complexity before streaming requirements are demonstrated. It remains a migration option if measured workloads justify it.

### Unix domain sockets and Windows named pipes

Reduce TCP exposure and provide strong local semantics, but create platform-specific endpoint, permissions, tooling, and packaging paths. They may be reconsidered if loopback threat modeling or platform requirements demand them.

### Fixed TCP port

Simplifies discovery but causes conflicts, complicates multiple instances, and creates a stable target for unintended clients. Rejected.

### Port discovery through a temporary file

Works across processes but introduces permissions, stale-file, cleanup, and race concerns. Structured `stdout` readiness is simpler under direct parent supervision.

### Authentication via environment variable or CLI argument

Environment inheritance and process-list visibility create unnecessary exposure. `stdin` avoids persistence and common process inspection surfaces.

### No authentication on loopback

Rejected because unrelated local processes could invoke the engine.

### OpenAPI as the initial source of truth

Combines HTTP operations and schemas and offers broad tooling. JSON Schema was selected because the immediate boundary needs language-neutral payload validation and the HTTP surface is small. Maintaining both manually would create drift.

### Manually maintained Go and TypeScript types

Fast initially but likely to diverge across runtimes and omit validation. Rejected as an authoritative contract strategy.

## Consequences

### Positive

- Straightforward Go and TypeScript implementation and debugging.
- Health and early routes can be exercised independently with authenticated test clients.
- No fixed-port conflict and support for multiple application instances.
- Secrets remain outside CLI arguments, files, renderer state, and persistent storage.
- JSON Schema supports language-neutral documentation, validation, and generation.
- The protocol remains simple while preserving a future migration path.

### Negative

- A local TCP listener still increases attack surface and requires careful authentication.
- Bearer-token lifecycle, bootstrap framing, and stdout/stderr discipline require integration tests.
- JSON Schema does not specify HTTP behavior by itself.
- Code-generation tooling and compatibility automation remain to be selected.
- Bounded pagination may not satisfy future streaming or export workloads.

## Required Verification

Before Phase 3 is considered complete, automated tests must prove:

- binding occurs only on loopback and uses an ephemeral port;
- malformed or absent bootstrap input prevents readiness;
- unauthenticated and incorrect-token requests are rejected;
- tokens and credentials do not appear in logs or renderer-facing errors;
- startup timeout, malformed readiness, incompatible versions, and unexpected exit fail safely;
- cancellation reaches a representative long-running Go operation;
- graceful shutdown drains or cancels according to policy and leaves no orphan process;
- packaged and development builds locate and supervise the correct engine binary.
